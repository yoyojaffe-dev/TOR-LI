## 3. Observed Requirements (EARS)

### 3.1 Barbershop discovery (read)
- The API **shall** expose `GET /barbershops?lat&lng&radius` returning barbershops within `radius` metres, nearest first. `barbershops.py:17`.
- The system **shall** constrain `radius` to `1..50000`; out-of-range **shall** return HTTP 422. `barbershops.py:21`.
- While querying, the system **shall** return only rows whose `place_type IN ('barber_shop','hair_care')`. `…place_type_photos_rating.sql` (`barbershops_within_radius`).
- When the radius RPC raises, the system **shall** return HTTP 502. `barbershops.py:29-30`.
- When a barbershop id is not found, `GET /barbershops/{id}` **shall** return HTTP 404. `barbershops.py:43`.

### 3.2 Slots
- The API **shall** expose `GET /slots?barbershop_id&only_free` returning slots ordered by `slot_time`; when `only_free` (default true), only `status='free'`. `slots.py:20-36`. **Note (QA): no time filter — past-dated free slots are returned.**
- The API **shall** expose `GET /slots/nearby?lat&lng&radius&limit` returning **free, upcoming** slots (`slot_time >= now()`) joined to shop info, nearest first. `slots.py:40`, `…reviews_and_nearby_slots.sql` (`available_slots_nearby`).
- The system **shall** bound `radius` `1..50000` and `limit` `1..100`; violations **shall** return 422; RPC failure **shall** return 502. `slots.py:42-52`.
- The API **shall** expose `GET /slots/realtime-info` describing the Realtime channel/table for clients. `slots.py:56`.

### 3.3 Pessimistic booking lock (the core invariant)
- When `POST /bookings/lock` is called, the system **shall** atomically set the slot to `locked` with `locked_until = now()+TTL` only if it is `free`, its lock is expired, OR the caller already holds it (re-entrant). `init.sql` `lock_slot`, `bookings.py:46`.
- While a slot is `locked` by another user with a live `locked_until`, the system **shall** reject a new lock and return HTTP 409. `lock_slot` else-branch; `bookings.py:50-52`.
- Default lock TTL **shall** be 300 seconds. `config.py:48` (`slot_lock_ttl_seconds`).
- When `POST /bookings/release` is called by the lock holder, the system **shall** free the slot; a non-holder **shall** get `success=false`. `release_slot`, `bookings.py:55`.
- When `POST /bookings/confirm` is called, the system **shall** invoke the Booking Agent; if it fails, the system **shall** release the lock and return HTTP 502. `bookings.py:61-74`.
- When confirming, the slot **shall** flip to `booked` only if the caller still holds a non-expired lock; otherwise the RPC **shall** report failure. `confirm_booking` (`…persist_bookings.sql`).
- On confirm, the system **shall** insert a `bookings` row carrying `user_token` + customer details. `…persist_bookings.sql`.

### 3.4 Booking history, cancel
- The API **shall** expose `GET /bookings?user_token` returning the caller's bookings joined with slot + shop (newest slot first). `bookings.py:29`, `bookings_for_user` RPC.
- When `POST /bookings/cancel` is called for a `confirmed` booking owned by `user_token`, the system **shall** mark it `cancelled` and free the slot; otherwise return HTTP 409. `bookings.py:37`, `…cancel_booking.sql`.

### 3.5 Reviews
- When `POST /reviews` is called, the system **shall** accept `rating` only in `1..5` (else 422) and persist a review **only if the booking belongs to the caller** (else HTTP 409). `reviews.py:18`, `submit_review` RPC, `schemas.py` `ReviewRequest`.
- A booking **shall** have at most one review (upsert on `booking_id`). `…reviews_and_nearby_slots.sql` (unique constraint).
- The API **shall** expose `GET /reviews?barbershop_id` returning recent reviews with a **masked** display name (first initial only). `reviews.py:27`, `reviews_for_barbershop`.

### 3.6 Agents (background)
- The Discovery Agent **shall** query Google Places for types `["barber_shop","hair_care"]`, paginate, dedupe by `place_id`, and upsert via `upsert_barbershop` (storing up to 6 photo URLs, rating, place_type). `discovery_agent.py:25`, `:_upsert`.
- The Scraping Agent **shall**, per pass, fetch scrappable shops, load each with headless Playwright, extract slots via OpenAI function-calling, and upsert via `upsert_free_slot`. `scraping_agent.py:204`.
- The Scraping Agent **shall** process shops concurrently bounded by `_MAX_CONCURRENT_SHOPS=5`. `scraping_agent.py:43,run_once`.
- `upsert_free_slot` **shall** update price only while a slot is `free`, never resetting `locked`/`booked` slots. `…upsert_free_slot.sql`.
- The Scraping Agent **shall** skip social/auth-walled domains (facebook/instagram/twitter/t.me). `scraping_agent.py:42,74`.
- The Booking Agent `submit` **shall** currently return a stubbed success (`stub: true`) — real Playwright submission not implemented. `booking_agent.py:25-41`.

### 3.7 Consumer client behavior
- On first run (no `localStorage.torli_onboarded`), the app **shall** redirect to the splash/onboarding flow. `app.js` init gate.
- The app **shall** generate + persist a per-browser `torli_user_token` (uuid) used for locks/bookings/reviews. `state.js` `getUserToken`.
- While a slot lock is held, the client **shall** show a live countdown and auto-release on expiry. `booking.js:14-29`.
- Where geolocation is denied, the app **shall** fall back to Jerusalem (31.7683, 35.2137). `app.js` `DEFAULT_POSITION`.

### 3.8 Barber dashboard
- The dashboard **shall** read `?shop=<uuid>` from the URL and list that shop's slots, live-updating via Realtime; missing param **shall** show an instruction message. `dashboard/js/app.js:24-46`.

## 4. Non-Functional Observations

- **Security (RLS):** `barbershops`/`available_slots` are world-readable (`anon`); owner-scoped writes gated by `is_shop_owner()` predicate. `…rls_policies.sql`, `…owner_scoped_writes_and_appointments.sql:19`.
- **Security (keys):** service-role key bypasses RLS and is restricted to agents; anon key is public-by-design (frontend). `supabase_client.py`, `config.py:27-28`.
- **Security (review ownership):** enforced in SQL (`submit_review` joins booking→user_token), proven by integration test. `tests/test_reviews_integration.py`.
- **Concurrency:** booking safety is enforced in Postgres (atomic `UPDATE … WHERE status=...`), not app code — race-proof. `init.sql lock_slot`.
- **CORS:** open (`allow_origins=["*"]`) — flagged for production. `main.py:32-37`.
- **Type/quality gates:** `mypy --strict` clean, ruff+black configured, branch-coverage gate `fail_under=90`. `backend/pyproject.toml`.

## 5. Inferred Acceptance Criteria

1. Two concurrent users cannot both confirm the same slot (second gets 409 on lock or fails confirm).
2. A user can only see/cancel/review their own bookings (scoped by `user_token`).
3. Non-barbershop POIs never appear in `/barbershops` results (place_type filter).
4. Scraping never clobbers a slot a user is mid-booking (`upsert_free_slot` free-only guard).
5. `/slots/nearby` only surfaces future, free slots.

## 6. Uncertainties & Questions (Flag)

1. **⚠️ Admin router mounting mismatch:** `admin.py:1-4` docstring says "Mounted only when ENVIRONMENT != production", but `main.py:45` mounts it **unconditionally** with no env guard. Either the docstring is stale or a guard is missing — **security-relevant** (admin endpoints trigger billed Google/OpenAI work). Needs decision.
2. **Booking Agent is a stub** — `/bookings/confirm` reports success without performing a real reservation on the barber's site. Is the booking considered "real" downstream? `booking_agent.py`.
3. **`list_slots` returns past slots** — intentional (fallback) or should it filter `slot_time >= now()`? Affects barber-profile slot lists. `slots.py:20`.
4. **`appointments` table vs `bookings`** — two booking representations coexist (consumer `bookings` by `user_token`; barber-side `appointments` by client/auth user). Relationship/migration path unclear. `…owner_scoped_writes_and_appointments.sql:108`.
5. **Auth not wired** — RLS owner policies assume an authenticated `auth.uid()`, but the consumer app uses an anonymous `user_token` and the dashboard takes `?shop=` from the URL. The authenticated barber flow appears designed but not connected.
6. **Twilio/SMS + payments** — settings exist (`config.py:35-40`, `payment_api_key`) but no runtime usage found in `app/` (SMS work was parked per project memory). Payments are UI mocks only.
7. **Duplicate-timestamp migrations** — two `20260626130000_*` files; apply order between them is ambiguous.

---

## 7. Recommendations

1. **Resolve admin mounting** — gate `app.include_router(admin.router)` behind `settings.environment != "production"` (or add auth), to match the docstring and protect billed agent triggers.
2. **Decide `list_slots` time semantics** — add `slot_time >= now()` filter (or a `future_only` flag) so barber profiles don't show stale slots.
3. **Finish or clearly mark the Booking Agent** — current stub makes confirm a false positive; surface "pending real submission" or implement Playwright path.
4. **Unify booking models** — document/ reconcile `bookings` (anon `user_token`) vs `appointments` (authed) before building the barber dashboard further.
5. **Tighten CORS** to the deployed origins before production.
6. **Rename duplicate-timestamp migration** to make ordering deterministic.
7. **Wire auth** (Supabase Auth) so owner-scoped RLS policies are actually exercised by the dashboard.
